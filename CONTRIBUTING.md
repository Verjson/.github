# Contributing to verJSON

First off — thank you for taking the time to contribute. 🙏

verJSON is an AI R&D accelerator: we research how AI can improve the way technology
products are built and delivered. Contributions of all kinds are welcome — code, docs,
ideas, and feedback.

These are the default contribution guidelines for all [Verjson](https://github.com/Verjson)
repositories. An individual project may add its own `CONTRIBUTING.md` that takes precedence.

## Code of Conduct

This project and everyone participating in it is governed by our
[Code of Conduct](./CODE_OF_CONDUCT.md). By participating, you are expected to uphold it.

## How can I contribute?

### Reporting bugs

- Search [existing issues](https://github.com/Verjson) first to avoid duplicates.
- Open a new issue using the **Bug Report** template and fill in as much detail as you can.

### Suggesting enhancements

- Open an issue using the **Feature Request** template.
- Explain the problem you're trying to solve, not just the solution you have in mind.

### Asking questions

- Use [Discussions](https://github.com/orgs/Verjson/discussions) for questions and ideas
  rather than opening an issue.

## Development workflow

1. **Fork** the repository and create your branch from the default branch.
2. **Make your changes** in small, focused commits with clear messages.
3. **Test** your changes and make sure existing tests pass.
4. **Update documentation** if your change affects behavior or usage.
5. **Open a pull request** using the PR template and link any related issues.

### Commit messages

Write clear, descriptive commit messages. We recommend the
[Conventional Commits](https://www.conventionalcommits.org/) style where practical:

```
feat: add support for X
fix: handle empty input in Y
docs: clarify setup instructions
```

### Pull requests

- Keep PRs focused — one logical change per PR is easier to review.
- Fill out the PR template completely.
- Make sure CI passes before requesting review.
- Be responsive to review feedback; we aim to review promptly in return.

## Reporting security issues

**Do not** open public issues for security vulnerabilities. Please follow our
[Security Policy](./SECURITY.md) instead.
